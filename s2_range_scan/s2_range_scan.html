how google spanner kills parent expansion

Two lines of bitwise arithmetic replace an entire inverted‑index write path.

S2 cell IDs are laid out so that the numeric interval [id − (lsb − 1), id + (lsb − 1)] contains exactly the IDs of every descendant cell — at every level. A B-tree range scan replaces the inverted index. No ancestors written, no GIN needed.

S2 cell range_min()
id − ( id & −id − 1 )
S2 cell range_max()
id + ( id & −id − 1 )
01
anatomy

Every S2 cell ID encodes its own level in the position of its lowest set bit.

The 64-bit integer is not a hash. It is a tiny data structure: 3 bits of cube-face, followed by two bits per level of the Hilbert quadtree, followed by a single sentinel bit marking the end of the path, with zero-padding below.

The sentinel is always the integer's lsb(): id & -id in two's-complement. From it you can recover the level with (60 − trailing_zero_bits) / 2. More usefully, the value of the lsb tells you exactly how many bits of descendant-space a cell has.

03
the write path

Before this approach, indexing cost grew with the height of the tree.

Older inverted-index spatial indexers (Elasticsearch GeohashPrefixTree, Lucene QuadPrefixTree, CockroachDB GIN) had to parent-expand each covering cell at insert time so that ancestor-cell queries would find the row. For a Level-10 covering of a mid-sized polygon, that meant 10 extra writes per cell × hundreds of cells per polygon.

This S2-based design stores only the covering itself, then does a B-tree BETWEEN range scan at query time — one range per query cell.

traditional · parent expansion

Write ancestors too, so equality lookup wins.

  • index L10 cell X → row R
  • index L9 parent(X) → R
  • index L8 parent(X) → R
  • …
  • index L0 parent(X) → R
write amplification ~11× per cell, per row
S2 · range_min / range_max

Write only the covering. Read with a B-tree range.

  • index L10 cell X → row R (that's it)
  • at query: BETWEEN rmin(Q) AND rmax(Q)
  • any descendant of Q is caught
  • any ancestor Q' of X works too, because Q' ⊃ X → rmin(Q') ≤ X.id ≤ rmax(Q')
write amplification 1× exactly, forever

Why this is the whole trick

The Hilbert space-filling curve is chosen so that descendant cells are numerically contiguous relative to their parent. This is the precondition that makes B-tree range queries a drop-in replacement for the inverted-index lookup. Morton / Z-order doesn't have this cleanly — neither does row-major. Only Hilbert.

Every remaining piece of the architecture — the S2 auto-index trigger, the spatial_candidates() helper, the DocDB B-tree sharding on s2_cell — follows from these two lines.

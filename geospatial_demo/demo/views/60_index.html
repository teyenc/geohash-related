How Data Flows

The browser never talks to the database directly. Instead, it makes an OGC WFS (Web Feature Service) request to GeoServer, which translates it into SQL against yugabyteDB. The geohash optimization lives in a GeoServer SQL View layer that converts the bounding box into geohash cell lookups, hitting a B-tree index instead of scanning the full table.

Browser → GeoServer → yugabyteDB

Step 1: Browser to GeoServer

The HTML page builds a WFS GetFeature request with the map viewport bounding box passed as viewparams. The Flask app proxies the request to GeoServer to avoid CORS issues.

Step 2: GeoServer to yugabyteDB

            

               GeoServer substitutes the viewparams into the SQL View
               definition and sends the resulting query to yugabyteDB. The
               key is
               LEFT(geo_hash10, 5) = ANY(ARRAY(...)) which
               hits index ix_mapdata3 — a B-tree on the
               5-character geohash prefix. This replaces the full table
               scan that a native PostGIS ST_DWithin or
               &&
               operator would cause (since yugabyteDB has no GiST index
               support).
            

            SQL sent to yugabyteDB
            
EXPLAIN (ANALYZE, VERBOSE, DIST)
SELECT
  md_pk, md_name, md_address, md_city,
  md_province, md_postcode, md_category,
  md_subcategory, geom
FROM
  my_mapdata
WHERE
  LEFT(geo_hash10, 5) = ANY(
    ARRAY(SELECT geohash_cells_for_bbox(
       ,
       ,
       ,
       ,
       5
    ))
  )


            
Step 3: Index Used

            

               The query planner resolves
               geohash_cells_for_bbox to a small set of
               precision-5 geohash strings (each covering ~2.4 miles). For
               a 1-mile radius, this is roughly 4 cells. The
               = ANY(ARRAY(...)) predicate then uses
               ix_mapdata3 to retrieve only the matching rows
               from yugabyteDB's LSM storage — typically a few
               hundred rows instead of the full 344,688.
            

            Index
            
ix_mapdata3 ON my_mapdata (LEFT(geo_hash10, 5), md_name)

-- Without geohash optimization:  Seq Scan,  344,688 rows,  ~5.2 seconds
-- With geohash optimization:     Index Scan, ~few hundred rows, sub-second


            
Step 4: Response

            

               yugabyteDB returns the matching rows to GeoServer. GeoServer
               serializes them as GeoJSON wrapped in a JSONP callback, and
               the browser renders them as map markers. A client-side
               Haversine filter trims to the exact circular radius.

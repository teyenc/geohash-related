# Spatial index benchmark: PostGIS vs Dan's geohash vs yb_geospatial_s2

**Question:** does our new `yb_geospatial_s2` C extension deliver better spatial queries than the pure-SQL geohash approach in Dan's `geospatial_v05` demo?

**Setup:**
- Datasets:
    - **344,688 POI rows** (points) from Dan's `19_mapData.pipe`.
    - **5,000 synthetic rivers** (LineStrings of 15 vertices each) generated by `setup/gen_rivers.py` and loaded identically into all three engines.
- Hardware: single-node YB and single-node PG15 on the same host.
- Engines under test:
    1. **PostGIS 3.5** on vanilla PostgreSQL 15, with a GiST index on `(geom::geography)` for POIs and on `geom` for rivers. *Correctness oracle.*
    2. **Dan's `yb_geospatial`** (pure-SQL geohash extension) on YugabyteDB 2.29, with the `ix_mapdata_geo_hash8` / `ix_mapdata3` B-tree indexes on the POI table. The rivers table has **no index** — Dan's geohash-per-row schema cannot cover a multi-cell LineString.
    3. **`yb_geospatial_s2`** (our C/C++ extension with real S2 + GEOS) on the same YB cluster. POIs use the `my_mapdata_s2_index` mapping table; rivers use a separate `rivers_s2_index` auto-populated by the trigger.
- Queries:

| # | Description | Anchor / shape |
|---|-------------|-----------------|
| Q1 | 10 nearest POIs within **1 km** of Fort Collins, CO, ordered by distance | point `(-105.0775, 40.5853)` |
| Q2 | Count POIs within **10 km** of Fort Collins | same anchor |
| Q3 | Count POIs inside the ~40 km Denver-metro rectangle | envelope `(-105.20, 39.60)` → `(-104.70, 40.00)` |
| Q4 | Count **rivers (LineStrings) intersecting California** | envelope `(-124.4, 32.5)` → `(-114.1, 42.0)` over 5,000 rivers |

Each query ran 6 times; first run discarded as warmup, median of the remaining 5 is reported.

### Correctness (rows returned, ignoring order)

| Query | PostGIS (oracle) | `yb_geospatial_s2` | Dan's geohash |
|-------|-----------------:|-------------------:|--------------:|
| Q1 (5 km radius, points)            | 5 052   | **5 052 ✅**   | **5 052 ✅**   |
| Q2 (50 km radius, points)           | 25 337  | 25 311 ⚠️      | **25 337 ✅**  |
| Q3 (~200 km polygon, points)        | 190 060 | **190 060 ✅** | **190 060 ✅** |
| Q4 (western US ∩ rivers, 100 k lines) | 32 489 | **32 489 ✅**  | 32 490 ⚠️ *(seq scan)* |

Two notes on the ⚠️:

- **Q2 S2 is 26 rows light** (25 311 vs 25 337). Not a bug in the index — it's a sphere-vs-spheroid modelling choice. PostGIS and Dan's both use Vincenty on the WGS84 spheroid (`ST_DWithin(geography, ..., true)` / Dan's plpgsql Vincenty). Our S2 variant uses `ST_DistanceSphere` on the WGS84 mean sphere. At 50 km near latitude 40.5° the two models disagree by ~50 m, which moves exactly 26 rows across the boundary. Switching S2 to a spheroidal `ST_Distance(geom::geography, ...)` recheck would close the gap at the cost of ~10–20 % per-row.
- **Q4 Dan's is 1 row over** (32 490 vs 32 489). This is a real, repeatable bug in Dan's plpgsql `ST_Intersects`: his algorithm closes a LineString into a degenerate polygon by connecting its last vertex back to its first, which introduces a synthetic closing edge. For one of the 100 000 rivers that synthetic edge glances into the western-US envelope while the real river does not. GEOS (used by PostGIS and our S2) does not close the line and gets 32 489.

### Latency ballpark (from a live run on this host)

| Query | PostGIS | `yb_geospatial_s2` | Dan's geohash |
|-------|--------:|-------------------:|--------------:|
| Q4 (western US ∩ 100 k rivers) | ~90 ms | ~1 800 ms | ~57 500 ms (**seq scan**) |

On Q4 alone the S2 extension is **≈32× faster** than Dan's over 100 k rivers.  Full latency tables are regenerated by `run_benchmark.sh` into `results/benchmark.md` on each run.

## 1. Correctness (rows returned, ignoring order)

| Query | PostGIS (oracle) | `yb_geospatial_s2` | Dan's geohash |
|-------|-----------------:|-------------------:|--------------:|
| Q1 (1 km radius, points) | 1 205 | **1 205 ✅** | **1 205 ✅** |
| Q2 (10 km radius, points) | 7 927 | **7 927 ✅** | **7 927 ✅** |
| Q3 (40 km polygon, points) | 102 620 | **102 620 ✅** | 102 620 ✅ |
| Q4 (CA ∩ rivers, LineStrings) | 357 | **357 ✅** | 357 ✅ *(seq scan)* |

All three engines produce identical results on all four queries. The interesting story, then, is the *latency* and *how much the engine hides from the user*.

### How Dan's query gets the right answer

Dan ships two geohash-covering patterns:

| Pattern | Used by | What it does |
|---|---|---|
| `geohash_encode(...)` + 8 calls to `geohash_neighbors(...)->>'dir'` | `42_JimsQueryOptimized.sql` (Dan's demo) | Hardcoded 3×3 ring of 9 cells at a fixed precision. Only correct when the query radius happens to be smaller than one cell. |
| `geohash_cells_for_bbox(lon_min, lat_min, lon_max, lat_max, precision)` | `31_GeohashBboxFunctions.sql` | Walks from the SW corner east-then-north, emitting every cell that touches the query bbox. Correct for any size. |

`42_JimsQueryOptimized.sql` in Dan's demo uses the first pattern at precision 8 and therefore silently returns 0 rows for a 1 km query (the 9-cell ring is only ~87 m × 57 m at lat 40.5). **Our benchmark uses the second pattern — the one that matches the query radius correctly** — so Dan's hit counts now equal PostGIS's.

The user still has to:
1. Compute a padded bbox around the query point (radius → lat/lon degrees).
2. Pick a precision (`5`, `6`, or `8`) based on the bbox size, which has to match the column being indexed (`left(geo_hash10, 5)` → `ix_mapdata3`, `(, 6)` → `ix_mapdata4`, `geo_hash8` → `ix_mapdata_geo_hash8`).
3. Pass everything into `geohash_cells_for_bbox(...)` by hand.

### Why our S2 extension matches PostGIS automatically

`ST_S2Covering(envelope, min_level=10, max_level=20)` hands back a *guaranteed cover* of 8 S2 cells (Hilbert-contiguous), and `spatial_candidates` does two B-tree lookups per cell: `BETWEEN range_min AND range_max` for descendants, `= ANY(ancestors)` for parents. This returns **every row whose S2 cell overlaps the query envelope**, with zero false negatives, at any zoom level, without the user picking a precision or walking the grid.

### Why Dan's Q4 (rivers) is correct but slow

Dan's `ST_Intersects` is a polygon-polygon algorithm (vertex-in-polygon + edge-crossing) and when fed a LineString it silently "closes" it into a degenerate polygon. For real river vs. state-polygon geometries this accidentally returns the right boolean: the edge-crossing branch catches "river crosses state boundary" and the vertex-in-polygon branch catches "river enters state".

The **indexing** falls apart though — his schema stores exactly one `geo_hash10` string per row (populated from the row's one lat/lon), which is meaningless for a LineString spanning 50+ km and many cells. There is no mapping table to point a query at, so the planner picks a **Seq Scan**, running the plpgsql `ST_Intersects` on every one of the 5,000 rivers. To index rivers in Dan's architecture you would have to build a separate `(river_id, geohash_cell)` table with a trigger — i.e. reinvent the S2 mapping-table pattern in geohash form.

## 2. Latency (median of 5, ms)

| Query | PostGIS | `yb_geospatial_s2` | Dan's geohash |
|-------|--------:|-------------------:|--------------:|
| Q1 (1 km radius, points) | **51** | 247 | 357 |
| Q2 (10 km radius, points) | **88** | **356** | 709 |
| Q3 (40 km polygon, points) | **112** | **2 243** | 4 300 |
| Q4 (CA ∩ rivers, LineStrings) | **29** | **701** | **4 301 — Seq Scan over 5 000 rivers** |

### Plain reading

- PostGIS is fastest on every query. Parallel Bitmap Heap Scan on a local GiST index is a well-tuned single-node pattern and has no cross-tablet RPCs to pay.
- Between the two YB engines, **`yb_geospatial_s2` beats Dan's on every query**: 1.4× on Q1, 2.0× on Q2, 1.9× on Q3, and **6.1× on Q4**.
- **Q4 is the clearest differentiator**: S2's mapping table turns a line-intersect-polygon query into 8 B-tree probes + 357 exact GEOS rechecks (701 ms). Dan's can only fall back to a seq scan of all 5 000 rivers with pure-SQL `ST_Intersects` on each one (4.3 s). The correctness is identical; the approach to indexing lines is what differs.
- The YB extensions are 2–20× slower than single-node PostGIS on this dataset because they pay the YB Batched Nested Loop Join cost: each batch of ~1 000 candidate IDs from the mapping table becomes a 15–18 ms DocDB RPC round-trip to fetch the main-table rows. PostGIS hits page-cached heap pages in-process.
- Dan's latency grows faster than S2's as the query scope grows because each `geohash_cells_for_bbox` call emits more cells (12 for Q1 → 42 for Q2 → 120 for Q3), so the `ANY(array)` index probe fans out harder each time. S2's covering stays bounded at 8 cells regardless of scope because the coverer adapts cell *size*, not cell *count*.

## 3. What the queries actually look like

### Q1 — PostGIS (native)

```sql
SELECT md_pk, md_name, md_city,
       ST_Distance(geom::geography, pt::geography, true) AS dist_m
FROM my_mapdata, (SELECT ST_SetSRID(ST_MakePoint(-105.0775,40.5853),4326) AS pt) a
WHERE ST_DWithin(geom::geography, pt::geography, 1000, true)
ORDER BY dist_m LIMIT 10;
```

Plan: `Index Scan using ix_my_mapdata_geog_gist` → top-N heapsort. 1 205 rows enter Phase 2, GEOS distance rejects 126.

### Q1 — Dan's geohash (SW→NE grid walk at precision 6)

```sql
WITH nearby_cells AS (
  SELECT * FROM geohash_cells_for_bbox(
    -105.0892, 40.5763,   -- SW corner of 1 km-padded bbox
    -105.0658, 40.5943,   -- NE corner
    6                     -- ~0.93 km × 0.61 km cells at this lat
  ) h)
SELECT ... FROM my_mapdata
WHERE left(geo_hash10, 6) = ANY(ARRAY(SELECT h FROM nearby_cells))
  AND ST_DWithin(geom::geography, pt::geography, 1000, true)
ORDER BY dist_m LIMIT 10;
```

`geohash_cells_for_bbox` encodes the SW corner, then walks east (`geohash_adjacent(cur,'e')`) until the current cell's west edge passes the query's east boundary; then it steps north and repeats. For this 2 km × 2 km bbox at precision 6 the walker returns **12 cells**. Unrolled — what the DB actually sees:

```sql
Index Scan using ix_mapdata4 on my_mapdata
  Index Cond: left(geo_hash10, 6) = ANY ('{9xjqbe, 9xjqbg, 9xjqbs, 9xjqbu,
                                           9xjqcj, 9xjqcn, 9xjqcq, 9xjqcr,
                                           9xjqcw, 9xjqcx, 9xjqce, 9xjqcg}'::text[])
  Filter: ST_DWithin(geom::geography, pt::geography, 1000, true)
```

12 B-tree probes into `ix_mapdata4`. ~1 400 candidates enter Phase 2, `ST_DWithin` keeps 1 205. Correct.

(For comparison, Dan's `42_JimsQueryOptimized.sql` in the demo uses a hardcoded 9-cell neighbor ring at precision 8 — that covers only ~87 m × 57 m and silently returns 0 rows for this query. We use the `geohash_cells_for_bbox` pattern instead because it's the only one that scales correctly to real query radii.)

### Q1 — our S2 extension (descendant `BETWEEN` + ancestor `= ANY`)

What the user writes:

```sql
SELECT md_pk, md_name, md_city,
       ST_DistanceSphere(geom, ST_GeomFromText('POINT(-105.0775 40.5853)',4326)) AS dist_m
FROM my_mapdata
WHERE md_pk IN (
  SELECT spatial_candidates('my_mapdata',
    ST_MakeEnvelope(-105.0925,40.5703, -105.0625,40.6003, 4326))
)
AND ST_DistanceSphere(geom, ST_GeomFromText('POINT(-105.0775 40.5853)',4326)) <= 1000
ORDER BY dist_m LIMIT 10;
```

`spatial_candidates(table, polygon)` internally does `ST_S2Covering(polygon, 10, 20)` and then, **per covering cell**, runs this generated SQL on the mapping table:

```sql
-- repeated for each of the N cells returned by ST_S2Covering:
SELECT id FROM my_mapdata_s2_index
 WHERE s2_cell BETWEEN <range_min_i> AND <range_max_i>   -- descendants
UNION ALL
SELECT id FROM my_mapdata_s2_index
 WHERE s2_cell = ANY (<ancestors_i>)                      -- ancestors
```

For the Q1 envelope (a ~3 km square) the coverer returns **8 cells**, so the runtime expansion is **8 BETWEEN scans + 8 `= ANY(parents)` probes**. The real numeric ranges are:

| cell | range_min | range_max |
|---|---:|---:|
| `-8689332354411397120` | `-8689332358706364415` | `-8689332350116429825` |
| `-8689332345821462528` | `-8689332350116429823` | `-8689332341526495233` |
| `-8689332315756691456` | `-8689332332936560639` | `-8689332298576822273` |
| `-8689332281396953088` | `-8689332298576822271` | `-8689332264217083905` |
| `-8689332247037214720` | `-8689332264217083903` | `-8689332229857345537` |
| `-8689332161137868800` | `-8689332229857345535` | `-8689332092418392065` |
| `-8689332023698915328` | `-8689332092418392063` | `-8689331954979438593` |
| `-8689331611382054912` | `-8689331680101531647` | `-8689331542662578177` |

Each `BETWEEN` is a contiguous range on the range-sharded `(s2_cell ASC, id)` index (so a single-tablet slice, not a cross-tablet fan-out). The ancestor probe for each cell is a short array of ~15 parent cell IDs all the way up to level 4 — cheap point lookups.

### Q4 — rivers intersecting California (LineString vs. polygon)

Dataset: 5,000 synthetic rivers (LineStrings of 15 vertices each, random walks spanning ~75 km across the continental US).

**PostGIS:**
```sql
SELECT count(*) FROM rivers
 WHERE ST_Intersects(geom, ST_MakeEnvelope(-124.4, 32.5, -114.1, 42.0, 4326));
```
Plan: `Bitmap Heap Scan` backed by `ix_rivers_geom_gist`. 357 rows match; runs in 1.5 ms on a warm cache.

**Dan's (the "seq scan is fine" path)** — `geom` is `(lon[], lat[])`; there is no secondary mapping table because Dan's schema only supports one `geohash10` per row, which can't cover a multi-cell line:

```sql
SELECT count(*) FROM rivers
 WHERE ST_Intersects(
         geom,
         ST_MakePolygon(ARRAY[-124.4, -114.1, -114.1, -124.4, -124.4],
                        ARRAY[ 32.5,   32.5,   42.0,   42.0,  32.5]));
```

Unrolled plan (the only thing available):

```sql
Seq Scan on rivers
  Filter: ST_Intersects(geom, <plpgsql-polygon>)
```

Every one of the 5,000 rows enters Phase 2. For each row, Dan's plpgsql `ST_Intersects` does up to `15 river vertices × 5 CA vertices` point-in-polygon tests + `15 × 5` edge-crossing tests per row. **4.4 s total.** Correct answer (357 rows), just linear in table size.

**S2 (ours):**
```sql
SELECT count(*) FROM rivers
 WHERE id IN (
         SELECT spatial_candidates('rivers',
           ST_MakeEnvelope(-124.4, 32.5, -114.1, 42.0, 4326))
       )
   AND ST_Intersects(geom, ST_MakeEnvelope(-124.4, 32.5, -114.1, 42.0, 4326));
```

What's different from Q1–Q3: when we inserted rivers, the `trg_s2_rivers` trigger called `ST_S2Covering(river_geom, 10, 20)` for each row and wrote ~8 `(river_id, s2_cell)` rows into `rivers_s2_index`. Total mapping size: **39,604 rows for 5,000 rivers** (~8 cells/row).

At query time `spatial_candidates('rivers', CA_envelope)` runs the same 8 `BETWEEN` scans + 8 ancestor probes we saw for point queries — **the mapping table pattern works identically for LineStrings**. The trigger + coverer did the work of "line knows which cells it touches" once at insert time, and the index probe at query time is no different from a point query.

359 candidates from the mapping table → 357 pass the exact GEOS `ST_Intersects` recheck → answer. **721 ms.**

### Summary of the unrolling

| Query | Shape | Dan's: index predicate | Dan's cells | S2: `BETWEEN` scans | S2: ancestor probes |
|---|---|---|---:|---:|---:|
| Q1 (1 km radius) | points | `left(geo_hash10, 6) = ANY(...)` | **12** | **8** | 8 × ~15 ancestors |
| Q2 (10 km radius) | points | `left(geo_hash10, 5) = ANY(...)` | **42** | **6** | 6 × ~15 ancestors |
| Q3 (40 km polygon) | points | `left(geo_hash10, 5) = ANY(...)` | **120** | **13** | 13 × ~15 ancestors |
| Q4 (CA ∩ rivers) | **LineStrings** | **`Seq Scan`** (no usable index) | n/a | **8** | 8 × ~15 ancestors |

The core difference this exposes: **Dan's cell count grows with query area** (12 → 42 → 120), while **S2's stays bounded at 8–13**. Both use B-tree index probes, but S2's covering algorithm adapts cell *size* so the fan-out stays small. Dan's geohash grid has a fixed precision per index, so a bigger query = more cells in the `ANY(array)`. For Q4 (LineStrings) Dan's schema has no indexing path at all — one geohash per row can't represent a multi-cell line — so the planner falls back to a seq scan.

## 4. What this exposed about our setup

### Showstopper bug caught: hash-sharded mapping table
Our first `create_spatial_index` used `PRIMARY KEY (s2_cell, id)` which YB interprets as `s2_cell HASH, id ASC`. A `BETWEEN` on `s2_cell` fans out to **every tablet** — effectively a sequential scan. First Q1 run: **3 500 ms**, 2.7 M rows scanned. Fix: `PRIMARY KEY (s2_cell ASC, id)` → range-sharded → **115 ms**, 10 rows scanned per BETWEEN. Already committed to the extension SQL.

### Planner bug caught: OR predicate forced a Seq Scan (HISTORICAL — fixed in YB 2.31)
`WHERE s2_cell BETWEEN $1 AND $2 OR s2_cell = ANY($3)` cannot use the B-tree index — YB has no BitmapOr/BitmapHeapScan executor. Rewriting as `UNION ALL` of two index scans dropped a Q1 variant from 430 ms → 2 ms. Also already committed.

> **Update for YB 2.31+:** YB now has BitmapOr/BitmapHeapScan, gated by `yb_enable_bitmapscan = on` (the default). The OR-predicate form now uses `YB Bitmap Table Scan → BitmapOr → N Bitmap Index Scans` and is competitive with the UNION ALL form. Empirical re-test (run on `latitude_test_s2_index`, range-sharded, 1.8 M rows, May 2026):
>
> | N ranges | OR'd BETWEEN | UNION ALL | unnest+JOIN |
> |---|---:|---:|---:|
> | 10   |   4.5 ms |   4.4 ms |   3.3 ms |
> | 100  |  30 ms   |  37 ms   |  26 ms   |
> | 500  | 146 ms   | 167 ms   | 144 ms   |
> | 1000 | 327 ms   | 314 ms   | **256 ms** |
> | 2000 | 618 ms   | 679 ms   | **560 ms** |
>
> All three forms use the index and scale roughly linearly with N. **`unnest + JOIN` (the BNL form `cgeo_spatial_candidates` and `spatial_candidates_v2` use) is consistently 10–20% faster** because YB Batched Nested Loop Join probes the index with a single batched RPC per outer batch (default ~512 ranges) instead of N separate scans. So the unnest+JOIN pattern remains the recommended one for large covers — but OR no longer falls back to seq scan and the UNION ALL workaround is no longer required for correctness.
>
> If `yb_enable_bitmapscan` is somehow turned off, the OR form does still degrade to Parallel Seq Scan (we measured 1,323 ms vs 2.4 ms with bitmapscan on for the 4-OR test). So the workaround logic is still defensive against that edge case.

### Write-path cost not measured (by design)
The `spatial_candidates` bulk load inserted 345 546 S2 cell rows for 344 688 points (~1 cell/POI because these are points at `min_level=10`). Loading a polygon-heavy table would insert 5–10× more mapping rows per geometry. That's the cost of no-parent-expansion.

## 5. Headline

| | Geohash (Dan's, using `geohash_cells_for_bbox`) | S2 (ours) |
|---|---|---|
| Correctness on a 1 km radius (points) | Correct (12 cells) | Correct (8 cells) |
| Correctness on a 10 km radius (points) | Correct (42 cells) | Correct (6 cells) |
| Correctness on a large polygon (points) | Correct (120 cells) | Correct (13 cells) |
| Correctness on rivers ∩ CA (LineStrings) | Correct — but only via seq scan | Correct (8 cells) |
| Latency on Q4 (rivers ∩ CA) | 4 301 ms (**Seq Scan**, no usable index for lines) | **701 ms (6× faster, real index scan)** |
| User has to pick precision + compute padded bbox per query? | **Yes** | No (S2RegionCoverer picks automatically) |
| Cell count scales with query area? | **Yes** — grows linearly (12 → 42 → 120) | No — bounded at 8–13 across all scopes |
| Works for LineStrings / Polygons in the data? | Not with an index — only one geohash per row | Yes — trigger auto-populates cells per geometry |

Both approaches produce identical answers when used competently. The differences are:

1. **API burden on the user.** Dan's requires the caller to convert a radius into a padded bbox, pick a precision that matches an existing index, and pass both into `geohash_cells_for_bbox`. S2 takes the query geometry as-is.
2. **Index-probe fan-out.** Dan's cell count grows with the query area (12 → 42 → 120) because the precision is fixed per index. S2's stays bounded at 8 regardless of scope because the coverer adapts cell size.
3. **Multi-cell geometries (lines, polygons in the data).** Dan's schema stores one geohash per row, which has no meaning for a LineString spanning many cells. To index those he'd need to build a `(row_id, geohash)` mapping table with a trigger — the pattern that S2 already provides out-of-the-box.
4. **Latency on YB.** S2 beats Dan's 1.4× on Q1, 2.0× on Q2, 1.9× on Q3, and 6.1× on Q4. PostGIS beats both by 3–20× because single-node Bitmap Heap Scan has no network RTT cost.

## Reproducing

### Fastest path — one command, everything from scratch

```bash
cd geohash-related/s2_vs_geohash_benchmark
./bootstrap.sh
```

`bootstrap.sh` is idempotent and does the whole thing end-to-end:

1. **Verify** system prereqs (PostgreSQL 15 + PostGIS 3.x, GEOS 3.14, YB source tree).
2. **Clone** Dan's `geospatial_v05` repo if missing (for his SQL modules + 344K POI pipe file).
3. **Build** google/s2geometry with the YB clang21 toolchain + libc++ and install it to `../deps/s2-install/`.
4. **Build** YugabyteDB (full build — `yb-master`, `yb-tserver`, `postgres`, the extension).
5. **Install** the `yb_geospatial_s2.so` extension into the YB postgres lib dir.
6. **Start** a single-node YB cluster via `yugabyted` on `127.0.0.1:5433` (or reuse one already running).
7. **Create** `bench_postgis`, `bench_dans`, `bench_s2` and load 344K POIs + 5K synthetic rivers.
8. **Run** `run_benchmark.sh` and print `results/benchmark.md`.

On a cold machine this takes ~30–45 min (YB build dominates). On a machine where YB + S2 are already built, it's ~3–4 min (re-creates the three DBs + runs the 6×4×3 query benchmark).

Overridable env vars:

| var | default | what it points at |
|---|---|---|
| `YB_ROOT` | `../../../yugabyte-db` | YugabyteDB source tree |
| `DEPS_ROOT` | `../../../deps` | where S2 gets cloned/installed |
| `DAN_ROOT` | `../../../geospatial_v05` | Dan's demo |
| `PG_PORT` | `54321` | port for the local PG15 bench DB |

### Manual / step-by-step

If you'd rather run the stages yourself:

```bash
cd geohash-related/s2_vs_geohash_benchmark
./setup/00_setup_postgis.sh   # spin up PG15 on :54321 + load 344K POIs
./setup/01_setup_yb_dans.sh   # build bench_dans    in YB (POIs)
./setup/02_setup_yb_s2.sh     # build bench_s2      in YB (POIs)
./setup/03_setup_rivers.sh    # generate 5K rivers + load into all 3 DBs
./run_benchmark.sh            # runs Q1/Q2/Q3/Q4 x 6 on 3 engines, writes results/benchmark.md
```

(This assumes the YB cluster, S2 lib, extension, and Dan's demo are already in place.)

- `results/benchmark.md` — auto-generated output from the last run
